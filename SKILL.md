# Intern Annotation Ops Safe

这是这个仓库专用的公开版 `SKILL.md`。

它保留了原项目的核心业务逻辑，但去掉了真实姓名、真实负责人、真实路径和任何不适合公开展示的内容，方便你直接放到 GitHub、作品集或面试材料里使用。

## 适用场景

- 面试中展示自己如何把复杂业务流程抽象成模块
- 在 GitHub 上展示规则设计、流程产品化和业务系统化能力
- 给前端页面、Agent、Prompt 或脚本提供统一的公开版业务定义

## 脱敏边界

- 人员统一使用 `标注员01`、`标注员02`、`ANN-001`
- 负责人统一使用 `@项目负责人`
- 输出目录统一使用 `./demo-output/`
- 示例链接统一使用 `https://demo.example.com/...`
- 不出现真实姓名、真实群名、真实平台地址、真实本地路径

## 执行原则

- 不为继续执行而追问确认
- 一次性给出完整输出
- 数据缺失时使用默认值并继续
- 错误优先记录并继续
- 优先根据关键词自动路由

## 模块说明

### M01 发任务

输入：

- 任务描述
- 名单
- 截止时间
- 配额

输出一份适合群发的任务通知。

### M02 检查状态

输入：

- 标注者名单
- 已完成条数
- 配额

输出任务状态表和整体进度摘要。

### M03 催进度

根据延迟程度生成群提醒或单人提醒，语气保持业务协作感，不做攻击性表达。

### M04 汇总结果

基于试标或阶段性结果，统计总条数、冲突条数、混合极性条数和质量预警。

### M05 结算说明

根据周期、有效条数和单价，输出结构化结算说明。

### M06 筛选候选人

评分规则：

- 学历：本科 `18` / 硕士 `24` / 博士 `30`
- 标注经验：音乐标注 `40` / 通用标注 `24` / 无经验 `8`
- 可用时间：`>=20h` 为 `30`，`10~20h` 为 `20`，`<10h` 为 `10`

分层规则：

- A 级：`>= 75`
- B 级：`50 ~ 74`
- C 级：`< 50`

### M07 面试评价

评分维度：

- 音乐感知力：`35`
- 理解标注标准能力：`30`
- 认真度与可靠性：`20`
- 沟通协作：`15`

通过线：`60`

### M08 试标注筛选

输入字段：

- `annotator`
- `rule_type`
- `has_mixed_polarity`
- `dimension`
- `score`

计算规则：

```text
有效冲突数 = 纯冲突数 + 混合极性数 × 0.5
冲突率 = 有效冲突数 / 总条数
```

分档：

- A级：`< 15%`
- B级：`15% ~ 30%`
- C级：`> 30%`

### M09 入场通知

输出正式标注阶段的入场通知和基础说明。

### M10 培训通知

如果已有试标结果，优先围绕高冲突维度输出培训重点；如果没有，默认输出：

1. 词分一致性
2. 混合极性处理
3. 分数档位校准

### M11 月度汇总

默认顺序：

1. 汇总工作量
2. 调用评分模块
3. 更新画像
4. 生成月报

### M12 评分

```text
综合评分 = 质量分(50) + 数量分(30) + 及时率分(20)

质量分 = (1 - 冲突率) × 50
数量分 = MIN(完成条数 / 配额, 1.0) × 30
及时率分 = 按时提交次数 / 总提交次数 × 20
```

评级：

- S级：`>= 90`
- A级：`75 ~ 89`
- B级：`60 ~ 74`
- C级：`45 ~ 59`
- D级：`< 45`

### M13 自动筛选

默认决策：

- S/A：继续合作
- B：继续合作，但重点关注
- C：培训警告
- D：停止合作

### M14 动态画像

默认输出五块内容：

1. 基础背景
2. 能力特征
3. 行为特征
4. 风险特征
5. 当前优化方向

要求直接给结论，不展示推理过程；数据不足时标记 `[数据不足]`。

## 输出语气规范

### 开头

- 群发通知：`各位同学好，`
- 单人提醒：`Hi {姓名}，` 或 `@{姓名}`

### 结尾

- 常规：`感谢大家的配合。`
- 落款：`CC：@项目负责人`

### 表达要求

- 可以提醒、说明影响，但不要攻击性表达
- 不直接使用 `淘汰`、`开除`、`不合格`
- 优先用递进式表达，例如：
  - `如未按时完成，会影响后续任务分配`
  - `试标注的完成情况和质量会作为后续是否进入正式标注阶段的参考依据之一`
  - `对于在培训后仍存在较大理解偏差的情况，可能不再继续参与后续标注工作`

## 错误处理原则

| 场景 | 处理方式 |
|------|----------|
| 字段缺失 | 跳过该字段并标注已跳过 |
| 名单为空 | 输出占位符 |
| 日期解析失败 | 用当天日期替代 |
| 除零或 NaN | 输出 `0` 并标注数据不足 |
| 阈值为空 | 回退默认值 |
