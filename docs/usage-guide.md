# 使用指南

## 这份项目怎么用

这个目录可以当成一份公开版的 Skill 资产包来使用。

如果你的目标是：

- 面试演示
- GitHub 展示
- 给自己的 Agent / 自动化脚本提供业务定义底稿

那么优先看这三个文件：

1. `README.md`
2. `skill-definition.yaml`
3. `examples/`

## 推荐阅读顺序

### 1. 先看 `README.md`

快速理解这套 safe 项目是做什么的、为什么存在，以及它要体现什么能力。

### 2. 再看 `skill-definition.yaml`

这里是核心定义，包含：

- 路由关键词
- 默认参数
- 模块职责
- 评分和筛选规则
- 输出风格
- 错误处理

如果后续你要把它改造成真正的 Skill、Prompt 配置、前端规则文件或者后端配置文件，这个 YAML 可以直接作为基础稿。

### 3. 最后看 `examples/`

示例文件适合做三件事：

- 给面试官看“实际输出会长什么样”
- 给自己做表达演练
- 作为后续自动化输出模板的参考

## 如何继续扩展

这份 safe 版目录本身是静态定义，不绑定某个具体运行环境。后续可以往三个方向扩：

### 方向一：变成 Cursor Skill

可以把 `skill-definition.yaml` 的内容继续整理成 `SKILL.md` 或你自己的 Skill 格式，让 Agent 直接按模块执行。

### 方向二：变成前端配置

可以把模块、默认值、评分规则拆成前端可读取的 JSON/YAML，让页面从“写死逻辑”逐步转成“配置驱动”。

### 方向三：变成后端规则引擎

可以把筛选、评分、分层逻辑迁移到本地服务里，让 CSV 导入、统计、筛选和通知模板生成都通过接口完成。

## 面试时怎么讲更顺

推荐讲法分三步：

1. 先说问题本质：这不是单点功能，而是一整条流程很碎、很依赖人经验。
2. 再说你的方法：我把它拆成招聘、面试、试标、培训、评分、画像这些模块。
3. 最后说价值：这样既能支持实际协作，也能把经验沉淀成可复用规则。

## 公开展示时要注意什么

- 不要放真实姓名
- 不要放真实群消息截图
- 不要放真实平台地址
- 不要放真实本地路径
- 不要把本地映射表、内部数据表直接上传到公开仓库

## 一条实用建议

如果你后续还会继续做这个项目，建议把：

- `skill-definition.yaml`
- 前端页面里的规则逻辑
- README 里的公开口径

这三份内容始终保持一致。这样无论是面试演示、项目展示还是后续开发，口径都会比较稳。
